1. Konzept

RDF-Triple + OWL-Ontologie + Reasoner = der klassische Semantic-Web- Stack. Anders als UE3 (informeller Property-Graph) hat UE4 ein formal definiertes Vokabular: Klassen mit Hierarchie, Properties mit Domain/Range, Inverse-Relationen.

Der Vorteil: ein Reasoner kann implizite Fakten ableiten. Beispiel: wir definieren apple:CEO ⊑ apple:Executive. Eine Query auf alle Executives bekommt CEOs automatisch mit, ohne sie explizit zu listen. Das ist Reasoning.

Wann ist UE4 die richtige Wahl?

Wenn Fragen logische Schlüsse über Klassen-Hierarchien brauchen („alle Smartphones", „alle Personen mit Apple-Rolle"), oder wenn Daten aus mehreren strukturierten Quellen mit owl:sameAs verschmolzen werden sollen (UE3 + DBpedia).

3. Die Ontologie aufbauen

3.1

Repository in GraphDB anlegen

Über backend/data/graphdb_client.py::ensure_repository(): REST-POST mit multipart/form-data an /rest/repositories, Config-TTL beigelegt. Wichtige Felder: 

graphdb:base-URL "http://uc5.butscher.cloud/apple#"
graphdb:ruleset "owl-horst-optimized"
graphdb:disable-sameAs "false"   ← wichtig für DBpedia-sameAs!
graphdb:entity-index-size "10000000"
3.2

Ontologie-TTL definieren

data/migrations/graphdb/001_apple_ontology.ttl:

@prefix apple: <http://uc5.butscher.cloud/apple#> .
@prefix owl:   <http://www.w3.org/2002/07/owl#> .
@prefix rdfs:  <http://www.w3.org/2000/01/rdf-schema#> .

# Klassen-Hierarchie
apple:CEO        rdfs:subClassOf apple:Executive .
apple:Executive  rdfs:subClassOf apple:Employee .
apple:Employee   rdfs:subClassOf apple:Person .
apple:Smartphone rdfs:subClassOf apple:MobileDevice .
apple:MobileDevice rdfs:subClassOf apple:HardwareProduct .
apple:HardwareProduct rdfs:subClassOf apple:Product .

# Properties mit Inverse
apple:foundedBy       owl:inverseOf apple:founded .
apple:wasCEOOf        owl:inverseOf apple:hasCEO .
apple:predecessorOf   owl:inverseOf apple:successorOf .

# Property-Hierarchie
apple:wasCEOOf        rdfs:subPropertyOf apple:worksFor .

4. UE3 → GraphDB Bridge

4.1

Entitäten übersetzen

backend/ingest/ue4_ontology.py:

def _entity_uri(entity_key: str) -> str:
    # "PERSON:steve jobs" → "apple:SteveJobs"
    _type, name = entity_key.split(":", 1)
    s = unicodedata.normalize("NFKD", name)
    s = "".join(c for c in s if not unicodedata.combining(c))
    pascal = "".join(w[:1].upper() + w[1:] for w in s.split())
    return f"http://uc5.butscher.cloud/apple#{pascal}"

# INSERT in GraphDB
INSERT DATA {
  apple:SteveJobs a apple:Person ;
      rdfs:label "Steve Jobs"@de , "Steve Jobs"@en ;
      apple:description "Mitgründer von Apple …" .
}
4.2

Relationen mappen

UE3-Relations wie „GRUENDET" → apple:founded, „CEO_VON" → apple:wasCEOOf. Generisches fallback apple:associatedWith wenn kein Mapping greift.

5. DBpedia-Anreicherung

ModulWas holt es?
dbpedia_validator.py Personen mit dbo:founder/dbp:founders/dbo:keyPerson zu dbr:Apple_Inc. + Off-Topic-Personen zu UnrelatedPerson demoten
dbpedia_products.py 43 Apple-Produkte über dbo:manufacturer + Predecessor/Successor-Ketten
dbpedia_edges.py Cross-Edges zwischen UE3-Entities aus DBpedia (founded_by, designed_by, manufactures, manufacturedBy, ...)
Virtuoso-Gotcha

DBpedia läuft auf Virtuoso. dbr:Apple_Inc. als CURIE führt zum syntax error at '.' before 'dbo:founder' — der Punkt im PN_LOCAL wird als Triple-Terminator interpretiert. Lösung: volle IRI <http://dbpedia.org/resource/Apple_Inc.> in spitzen Klammern.

6. NL → SPARQL Pipeline

6.1

System-Prompt mit Datenrealität

Du erzeugst SPARQL-1.1-Queries für eine OWL-Ontologie über Apple
mit aktivem OWL-Horst-Reasoning.

DATENBEREINIGUNG:
- apple:UnrelatedPerson markiert Off-Topic-Personen.
  IMMER mit FILTER NOT EXISTS ausschließen.
- @de UND @en Labels → STR(?label) zum Deduplizieren.

QUERY-STRATEGIE-REGELN:
1. Wer war/ist X-Rolle:
   SELECT DISTINCT (STR(?l) AS ?name) WHERE {
     ?p a apple:CEO ; rdfs:label ?l .
     FILTER NOT EXISTS { ?p a apple:UnrelatedPerson }
   } LIMIT 30

2. Welche X-Produkte (Subklassen-Inferenz):
   ?x rdf:type/rdfs:subClassOf* apple:Smartphone ;
      rdfs:label ?n .

4. Produktchronologie:
   Für „Vorgänger von X":
     ?pred apple:predecessorOf ?x .
6.2

Postprocessing

Output durchläuft:

  • _strip_codefence() — entfernt ```sparql Wrap
  • _ensure_prefixes() — fügt fehlende PREFIX-Deklarationen automatisch an (auch wenn Modell sie vergisst)

7. Drei Fallback-Stufen

Phase 2 — primär
SPARQL gegen GraphDB
Wenn Bindings > 0 → fertig.
Phase 3 — Fallback 1
UE1 Text-Retrieval auf die Original-NL-Frage, wenn SPARQL 0 Bindings liefert und kein Syntax-Error vorlag. Chunks bekommen einen HINWEIS-Prefix („Ontologie-Query lieferte keine Treffer, folgende Wikipedia-Stellen sind relevant…")
Phase 4 — Fallback 2
Live-DBpedia-Lookup wenn auch UE1 dünn ist. extract_anchors(sparql) zieht die Entitätsnamen aus der SPARQL („PowerBook 145b"), fragt DBpedia live mit 8s-Timeout + 1 Retry. Treffer → formatierte Chunks mit Predecessor/Successor/ Manufacturer/Abstract.

8. Selbsttest

# 1. Container hochfahren
docker compose up -d

# 2. UE4-Daten initialisieren
docker compose exec app python -m backend.data.graphdb_client
# → lädt 001_apple_ontology.ttl, 002_seed_aliases.ttl,
#       003_canonical_persons.ttl

# 3. DBpedia-Anreicherungen (über API)
curl -X POST localhost:8000/api/ue4/validate
curl -X POST localhost:8000/api/ue4/enrich-products
curl -X POST localhost:8000/api/ue4/enrich-edges
curl -X POST localhost:8000/api/ue3/sync-canonical

# 4. Frage stellen
curl -X POST localhost:8000/api/query \
  -H "Content-Type: application/json" \
  -d '{"strategy":"ue4",
       "query":"Welche CEOs hat Apple gehabt?",
       "llm":"gemini"}'

# Erwartete Antwort: 5 Bindings (Jobs, Cook, Sculley, Spindler, Amelio)
# trace.sparql ist im Response → didaktische Demo

# 5. Frage die Live-DBpedia trigger:
curl -X POST localhost:8000/api/query \
  -H "Content-Type: application/json" \
  -d '{"strategy":"ue4",
       "query":"Was war vor dem PowerBook 145b?",
       "llm":"gemini"}'
# trace.live_dbpedia_used == true wenn DBpedia mitspielt
