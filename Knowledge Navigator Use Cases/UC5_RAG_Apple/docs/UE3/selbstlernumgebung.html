1. Konzept

Microsoft GraphRAG: Statt nur Volltextsuche bauen wir aus dem Korpus einen Wissensgraphen (Entitäten + Relationen), partitionieren ihn in Communities (Themengruppen) und können bei jeder Frage entweder lokal (an einer Entität) oder global (an einer Community) ansetzen.

Warum überhaupt einen Graph?

Eine Frage wie „Welche Personen waren an der Sculley-Ära beteiligt?" zerfällt in Volltextsuche zu einzelnen Erwähnungen. In einem Graph hängt man an „Sculley-Ära"-Knoten und folgt allen PARTICIPATED_IN-Kanten — strukturiert, deduplizierend.

3. Entity- und Relationsextraktion

3.1

Prompt-Design

Pro UE1-Chunk wird Gemini gebeten, alle Entitäten und Relationen als JSON zurückzugeben. Few-Shot Beispiele aus der Apple-Domäne im Prompt verbessern die Konsistenz. 

System: Du bist ein gründlicher NER-Extraktor für Apple-Wikipedia.

        Typen: PERSON, ORGANIZATION, PRODUCT, EVENT, LOCATION, CONCEPT.
        BESONDERS WICHTIG — historische CEOs werden oft übersehen,
        MÜSSEN aber extrahiert werden:

3.2

JSON-Parse + Resolution

  • Antwort wird mit Pydantic gegen strenges Schema gevalidet
  • Personen-Resolution: "Wozniak" → "Steve Wozniak" (Suffix-Match auf erstes Wort)
  • Apple-Konflikt: "Apple" (UE3) ↔ "Apple Inc." (DBpedia später via owl:sameAs)
  • Dedup nach (entity_type, name-lower)

4. Persistierung

4.1

Postgres-Schema

-- 007_ue3_entities_communities.sql
CREATE TABLE ue3.entity_summary (
  id            serial PRIMARY KEY,
  entity_key    text UNIQUE,    -- "PERSON:steve jobs"
  name          text NOT NULL,
  type          text NOT NULL,  -- PERSON|ORG|PRODUCT|EVENT|LOC|CONCEPT
  description   text,
  mention_count int DEFAULT 0
);
CREATE TABLE ue3.entity_mention (
  entity_key  text REFERENCES ue3.entity_summary(entity_key),
  chunk_id    int REFERENCES ue1.chunk(id),
  PRIMARY KEY (entity_key, chunk_id)
);
CREATE TABLE ue3.community_summary (
  community_id text PRIMARY KEY,
  level        int,
  size         int,
  summary      text,
  entity_keys  text[]
);
4.2

Neo4j-Knoten und -Kanten

// data/migrations/neo4j/001_constraints.cypher
CREATE CONSTRAINT entity_id IF NOT EXISTS
  FOR (e:Entity) REQUIRE e.id IS UNIQUE;

// Ingest erzeugt
MERGE (a:Entity {id: 'PERSON:steve jobs'})
  ON CREATE SET a.name='Steve Jobs', a.type='PERSON'
MERGE (b:Entity {id: 'ORGANIZATION:apple'})
  ON CREATE SET b.name='Apple', b.type='ORGANIZATION'
MERGE (a)-[r:RELATED_TO {type:'GRUENDET'}]->(b)
  ON CREATE SET r.weight=1
  ON MATCH  SET r.weight=r.weight+1;

5. Community-Detection (Louvain)

5.1

Algorithmus

Louvain optimiert die Modularität eines Graphen: Kantengewicht innerhalb der Cluster soll hoch sein, zwischen den Clustern niedrig. In Python: 

import networkx as nx
from community import community_louvain

G = nx.Graph()
for s, t, w in related_to_edges:
    G.add_edge(s, t, weight=w)

partition = community_louvain.best_partition(G)
# partition: {entity_key: community_id_int}
5.2

Community-Summaries

Pro Cluster die Members → Gemini → 2-Satz-Summary, gespeichert in ue3.community_summary. Verwendet die Mention-Chunks der Top-3 Members als Kontext.

6. Retrieval-Modi

6.1

local — Entity-anchored

def _local_retrieve(q_emb, keywords, type_hints, k):
    entities = match_entities(keywords, type_hints, q_emb)
    chunks = []
    for e in entities[:5]:
        mentions = pg.fetch_chunks_for(e.entity_key, limit=k)
        chunks.extend(mentions)
    return chunks, entities
6.2

global — Community-anchored

def _global_retrieve(q_emb, k):
    # Community summaries were embedded at ingest time
    comms = nearest_communities(q_emb, top=3)
    return [
        {"text": c.summary, "section_path": f"Community {c.id}",
         "kind": "community_summary"}
        for c in comms
    ], comms
6.3

hybrid (Default)

Beides parallel + dedup. Default-Modus, deckt sowohl spezifische Entitätsfragen als auch Übersichtsfragen ab.

8. Text-Fallback

Wenn UE3 nichts oder Dünnes findet, läuft UE1 als Augmentierung: 

any_canonical = any(e.key.startswith("CANON:") for e in matched_entities)
thin_match    = len(merged) < k // 2
no_terms      = not any(t in chunk.text.lower() for t in query_terms for chunk in merged)

if not merged or thin_match or any_canonical or no_terms:
    text_chunks = SimpleRAG.retrieve(query, k)
    chunks      = merged + dedupe_against(merged, text_chunks)

→ UE3 ist nie schlechter als UE1, oft besser (durch Struktur-Anker).
