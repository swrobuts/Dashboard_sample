Schrittweise: wie der UE1-Stack um eine LLM-summarisierte Sektions- Baumstruktur erweitert wird, damit Übersichts-Fragen punktgenau in der richtigen Sektion landen.

1. Konzept

Reine semantische Suche (UE1) findet einzelne Textstellen — gut für Punktfragen, schlecht für Übersichtsfragen. PageIndex fügt eine zusätzliche Schicht hinzu: eine Hierarchie von Sektions- Summaries, durch die ein LLM bei jeder Frage erst navigiert (wie ein Mensch durch ein Buch-Inhaltsverzeichnis), bevor die eigentliche Volltextsuche dispatcht wird.

Idee in einem Satz

Bevor du Sätze durchsuchst, lies das Inhaltsverzeichnis. Bei einem 500-seitigen Buch sparst du dir 99 % Lesezeit. Genau das macht PageIndex für RAG.

3. Den Tree bauen

3.1

Migration für die Baum-Tabelle

-- 005_ue2_tree.sql
CREATE TABLE ue2.tree_node (
  id          serial PRIMARY KEY,
  snapshot_id int REFERENCES raw.snapshot(id),
  parent_id   int REFERENCES ue2.tree_node(id),
  section_id  int REFERENCES clean.section(id),  -- NULL für Synth-Knoten
  path        text NOT NULL,
  depth       int NOT NULL,
  ordinal     int NOT NULL,
  summary     text NOT NULL,
  llm_model   text,
  created_at  timestamptz DEFAULT now(),
  UNIQUE (snapshot_id, path)
);
CREATE INDEX ON ue2.tree_node(parent_id);
CREATE INDEX ON ue2.tree_node(snapshot_id, depth);

4. Bottom-Up Summarization

4.1

Leaf-Sektionen zusammenfassen

Pro Leaf: alle UE1-Chunks dieser Sektion ans LLM mit Prompt:

System: Du fasst Wikipedia-Sektionen für eine Knowledge-Base zusammen.
        Max 3 Sätze, faktenfokussiert, keine Spekulation.
User:   Sektion: {path}
        Text: {chunks_joined}
        Fasse zusammen.
4.2

Eltern-Sektionen rekursiv

System: Du fasst eine Buchkapitel-Ebene zusammen, indem du die
        Unterkapitel-Summaries integrierst.
User:   Kapitel: {path}
        Unterkapitel:
          1. {child1.path} — {child1.summary}
          2. {child2.path} — {child2.summary}
        Fasse das Kapitel in 2-3 Sätzen zusammen.

→ Information fließt bottom-up: Leaf-Inhalt → Mid-Level Themes → Top-Level Strukturüberblick.

4.3

Speichern + Idempotenz

Jeder Knoten in ue2.tree_node ist eindeutig per (snapshot_id, path). Re-Run mit gleichem Snapshot ist idempotent (UPSERT).

5. Navigation zur Anfragezeit

5.1

Start: Wurzel

Die Wurzel-Summary beschreibt das ganze Lemma. Das LLM sieht alle Top-Level-Kinder mit ihren Kurz-Summaries.

5.2

Iterativ vertiefen

System: Du wählst aus einer Liste von Sektionen jene, die zur
        Beantwortung der Frage relevant sind.
User:   Aktuelle Sektion: {node.path}
        Summary: {node.summary}

        Kinder:
          1. Hauptsitz — Apple-HQ in Cupertino seit 1993 …
          2. Apple Stores — 16 Stores in Deutschland …
          3. Mitarbeiter — 164 000 weltweit Stand 2024 …

        Frage: "Wo hat Apple seinen Hauptsitz?"

        Antworte JSON: { "selected": [Indizes], "dive_deeper": bool }
5.3

Hallu-Schutz

Pydantic-Schema-Parse mit strikter Validation. Wenn das LLM einen Index nennt der nicht in der Kinderliste war: selected = [0] (erstes Kind), dive_deeper = False. Bestraft das Modell und vermeidet Crashes.

6. Filtered UE1 Retrieval

Wenn die Navigation Terminal-Sektionen ausgewählt hat, läuft die UE1-Pipeline mit einem zusätzlichen Filter: 

SELECT … FROM ue1.chunk
WHERE section_id IN (:selected_section_ids)
ORDER BY embedding <=> :q_emb LIMIT 30;

Der Rest (BM25, RRF, Reranker, MMR) läuft genau wie in UE1. Die Antwort referenziert die Section-Pfade wie zuvor.

7. Selbsttest

curl -X POST localhost:8000/api/ingest \
  -H "Content-Type: application/json" \
  -d '{"strategy":"ue2"}'  # generiert den Tree (dauert ca. 2–3 min)

curl -X POST localhost:8000/api/query \
  -H "Content-Type: application/json" \
  -d '{"strategy":"ue2","query":"Was war die Sculley-Ära?","llm":"gemini"}'

Im Frontend: UE2 wählen, Frage eingeben. Im Trace-Output siehst du den Navigations-Pfad: "path": ["Geschichte", "Sculley-Ära und danach"].
