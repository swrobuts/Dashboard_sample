UE1 — Simple RAG mit Hybrid-Retrieval

1. Konzept

RAG = Retrieval-Augmented Generation. Das LLM bekommt bei jeder Frage nur die wenigen Textstellen aus dem Korpus, die wirklich zur Frage passen, statt sich auf sein Trainingswissen zu verlassen. Das verhindert Halluzinationen und macht Antworten überprüfbar.

„Simple RAG" heißt: kein Wissens-Graph, keine Tree-Navigation, keine Ontologie. Nur Textstücke + Vektor-Index + Volltextindex + LLM.

Warum „Hybrid"?

Reine Vektor-Suche scheitert oft an Eigennamen und Zahlen („PowerBook 165", „740 Mio. Dollar"). Reine Stichwortsuche scheitert bei semantischen Fragen („Welche Krise hatte Apple?"). Wir kombinieren beide via Reciprocal Rank Fusion (RRF) — das Beste aus zwei Welten.

3. Datenmodell anlegen

Drei Schichten in Postgres — sauber getrennt nach Verantwortung:

3.1

Schema raw · clean · ue1

-- 001_extensions_and_meta.sql
CREATE EXTENSION IF NOT EXISTS vector;
CREATE SCHEMA raw;
CREATE SCHEMA clean;
CREATE SCHEMA ue1;
CREATE SCHEMA meta;

-- 002_raw_snapshot.sql
CREATE TABLE raw.snapshot (
  id          serial PRIMARY KEY,
  url         text NOT NULL,
  fetched_at  timestamptz DEFAULT now(),
  revision_id text,
  content     text,
  content_hash text
);

-- 003_clean_section.sql
CREATE TABLE clean.document (...);
CREATE TABLE clean.section (
  id         serial PRIMARY KEY,
  document_id int REFERENCES clean.document(id),
  parent_id  int REFERENCES clean.section(id),
  path       text NOT NULL,       -- "Geschichte/1976-1980: Gründung"
  depth      int NOT NULL,
  ordinal    int NOT NULL,
  body_md    text
);

-- 004_ue1_chunk.sql + 006_tsvector_gin.sql
CREATE TABLE ue1.chunk (
  id          serial PRIMARY KEY,
  section_id  int REFERENCES clean.section(id),
  ordinal     int NOT NULL,
  text        text NOT NULL,
  embedding   vector(768),
  text_tsv    tsvector
    GENERATED ALWAYS AS (to_tsvector('german', text)) STORED
);

CREATE INDEX ON ue1.chunk USING hnsw (embedding vector_cosine_ops)
  WITH (m=16, ef_construction=64);
CREATE INDEX ON ue1.chunk USING GIN (text_tsv);

4. Ingest-Pipeline

4.1

Wikipedia-Artikel laden

backend/data/wikipedia_loader.py — fetched die deutsche Wikipedia-Seite via httpx, speichert Roh-HTML in raw.snapshot mit Hash zur Idempotenz.

r = httpx.get(settings.wikipedia_url, timeout=30)
content = r.text
content_hash = sha256(content.encode()).hexdigest()
# upsert in raw.snapshot
4.2

HTML cleanen

backend/data/cleaner.py entfernt Infoboxen, Navigation, Referenzen, Edit-Links etc. Behandelt den modernen MediaWiki- <div class="mw-heading">-Wrapper korrekt.

def _is_heading_wrapper(tag):
    return tag.name == "div" and "mw-heading" in (tag.get("class") or [])
def _effective_start(h):
    # H2 inside mw-heading → next_siblings of WRAPPER, not h
    parent = h.find_parent(_is_heading_wrapper)
    return parent if parent else h
4.3

Sektionen extrahieren

Headings H2/H3 werden zu einer Baumstruktur in clean.section. Der Section-Pfad „Geschichte/1985–1996: Sculley-Ära" ist später beim Antworten als Zitat sichtbar.

4.4

Chunken mit Section-Prefix

backend/data/chunker.py teilt Markdown in Token-Budget- Chunks (~500 Tokens). Jeder Chunk bekommt seinen Section-Pfad vorangestellt — das verbessert die Embeddings dramatisch.

# Chunk-Text vor Embedding:
"[Geschichte/1976-1980: Gründung]
Steve Jobs, Steve Wozniak und Ronald Wayne
gründeten Apple am 1. April 1976 …"

→ Embedding kennt jetzt nicht nur den Inhalt, sondern auch worüber der Inhalt steht.

4.5

Embeddings holen

backend/llm/embeddings.py → Gemini gemini-embedding-001 mit output_dimensionality=768 (Matryoshka-Truncation des 3072-dim Vollformats) und manueller L2-Normalisierung.

resp = client.embed_content(
    model="gemini-embedding-001",
    content=batch_texts,
    output_dimensionality=768,
)
vecs = np.array(resp.embedding) / np.linalg.norm(...)  # L2-normalize
4.6

Schreiben + Index

Insert in ue1.chunk; text_tsv wird durch die GENERATED-Column automatisch gefüllt; HNSW-Index erlaubt schnelle Cosinus-Suche.

5. Retrieval-Pipeline

backend/retrieval/pipeline.py macht den Kern. Bei jeder Frage:

5.1

Dense-Suche (pgvector)

SELECT id, text, section_path,
       embedding <=> :q_emb AS distance
FROM ue1.chunk
ORDER BY distance LIMIT 30;
5.2

Sparse-Suche (BM25 via tsvector)

SELECT id, text, ts_rank_cd(text_tsv, q) AS rank
FROM ue1.chunk,
     to_tsquery('german', :keywords) AS q
WHERE text_tsv @@ q
ORDER BY rank DESC LIMIT 30;
5.3

Reciprocal Rank Fusion

Jeder Chunk bekommt eine Score basierend auf seinen Rängen in den beiden Listen: 

RRF(d) = Σᵢ 1 / (60 + rank_i(d))

Konstanter Offset 60 dämpft die Spitzen → Chunks die in beiden Listen mittel-hoch ranken gewinnen vor Chunks die in einer Liste ganz oben aber in der anderen gar nicht vorkommen.

5.4

LLM-Reranker

Top-15 Kandidaten gehen an Gemini mit dem Auftrag: „Für jeden Chunk: wie relevant zur Frage auf einer Skala 1–10?" Daraus die Top-K (default 8).

5.5

MMR-Diversifikation

Maximal Marginal Relevance: bei der Auswahl der finalen K Chunks wird Redundanz bestraft. λ steuert die Balance. 

MMR(d) = λ · sim(d, query) − (1−λ) · max sim(d, dⱼ)
                                        ⱼ∈selected

Ergebnis: K verschiedene Chunks, nicht 5× derselbe Absatz aus leicht unterschiedlichen Perspektiven.

6. Antwort-Generierung

Die finalen Chunks gehen mit der Originalfrage an Gemini 2.5 Flash: 

System: Beantworte die Frage AUSSCHLIESSLICH auf Basis
        der Auszüge. Zitiere jeden Fakt mit dem Section-Pfad
        in Klammern. Halluziniere nichts dazu.

Auszüge:
  [Geschichte/Gründung]
  Steve Jobs … gründete Apple am 1. April 1976 …

  [Geschichte/Sculley-Ära]
  John Sculley wurde 1983 CEO …

Frage: Wer hat Apple gegründet?
→ Antwort: "Apple wurde von Steve Jobs, Steve Wozniak und
   Ronald Wayne am 1. April 1976 gegründet (Geschichte/Gründung)."

7. Selbsttest

Nach dem Hochfahren des Docker-Compose-Stacks:

docker compose up -d
# Lege UE1 an:
curl -X POST localhost:8000/api/ingest \
  -H "Content-Type: application/json" \
  -d '{"strategy":"ue1"}'

# Frage stellen:
curl -X POST localhost:8000/api/query \
  -H "Content-Type: application/json" \
  -d '{"strategy":"ue1","query":"Wann wurde Apple gegründet?","llm":"gemini"}'

8. Wohin als nächstes

UE1 ist die Basis. Die nächsten Übungseinheiten erweitern:

  • UE2 (PageIndex) — fügt eine LLM-summarisierte Baum-Navigation hinzu für „Übersichtsfragen"
  • UE3 (GraphRAG) — extrahiert Entitäten/Relationen in einen Property-Graph (Neo4j) für strukturierte Fragen
  • UE4 (Ontology) — OWL-Ontologie in GraphDB + NL→SPARQL für Reasoning-Fragen
